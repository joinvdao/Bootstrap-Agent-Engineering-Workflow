# Public Ticketing

Public work is tracked with GitHub Issues and Pull Requests.

Repo-local sprint tickets may be used for detailed execution breakdowns when a goal needs multiple implementation steps. GitHub Issues remain the public roadmap and discussion surface; sprint tickets are the working plan for a specific sprint.

## Issue Types

- Bug reports: reproducible defects and regressions.
- Feature requests: public product or platform proposals.
- Engineering tasks: scoped implementation, cleanup, research, or verification work.
- Livestream follow-up: public notes and action items from community build sessions.

## Agent-Ready Issues

Good public issues include:

- Goal.
- Scope.
- Acceptance criteria.
- Validation plan.
- Relevant docs or files.
- Privacy/safety constraints.

Do not include private planning context, secrets, local paths, personal contact details, unpublished provider agreements, exact private addresses, or sensitive geospatial records.

## Pull Requests

Pull requests should link a public issue when possible, explain the change, list validation commands, and include a privacy check.

## Sprint Ticket Linkage

When using repo-local sprint tickets, link the sprint or step from the related issue or pull request. Keep ticket content public-safe and avoid duplicating sensitive context.
